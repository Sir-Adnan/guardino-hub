# API and Telegram Bot Readiness

Guardino Hub خودش ربات تلگرام نیست، اما API آن برای ساخت ربات فروش آماده شده است.

## مدل اتصال

ربات می‌تواند با یکی از این حالت‌ها به گاردینو وصل شود:

- token سوپرادمین برای مدیریت کل سیستم
- token رسیلر برای فروشگاه اختصاصی همان رسیلر

هر token باید فقط به محدوده همان حساب دسترسی داشته باشد. برای ربات فروش رسیلر، بهتر است token همان رسیلر استفاده شود تا ربات نتواند به کاربران یا موجودی رسیلرهای دیگر دسترسی پیدا کند.

## جریان خرید پیشنهادی

1. کاربر وارد ربات می‌شود.
2. تعرفه‌ها و نودهای مجاز از API خوانده می‌شوند.
3. کاربر پلن را انتخاب می‌کند.
4. پرداخت در ربات یا gateway انجام می‌شود.
5. ربات با `client_request_id` یکتا سفارش ساخت کاربر را در Guardino ثبت می‌کند.
6. Guardino کاربر را در پنل بالادستی می‌سازد، از کیف پول رسیلر کم می‌کند و لینک ساب را برمی‌گرداند.

`client_request_id` برای جلوگیری از دوباره‌سازی کاربر هنگام retry ضروری است.

## رسیلر با پنل اختصاصی روی نود مشترک

برای رسیلرهای بزرگ می‌توان روی یک نود PasarGuard یا Marzban، credential اختصاصی همان رسیلر را در تخصیص نود ثبت کرد. بعد از آن:

- کاربران جدید همان رسیلر با credential اختصاصی ساخته می‌شوند.
- کاربران قبلی آن credential قابل import هستند.
- import از کیف پول کم نمی‌کند.
- تمدید و افزایش حجم بعد از import طبق قوانین مالی Guardino محاسبه می‌شود.

## نکات امنیتی

- token رسیلر را در ربات همان رسیلر استفاده کنید، نه token سوپرادمین.
- tokenها باید قابل revoke باشند و در لاگ‌ها چاپ نشوند.
- برای عملیات مالی همیشه `client_request_id` یکتا بفرستید.
- ربات نباید خودش ledger را دور بزند؛ همه تغییرات مالی باید از API رسمی Guardino عبور کنند.

## endpointهای مهم

مسیرهای دقیق را از OpenAPI نصب‌شده بخوانید:

```text
https://YOUR_DOMAIN/api/openapi.json
https://YOUR_DOMAIN/api/docs
```

مسیرهای مهم برای ربات فروش معمولا شامل auth/token، لیست پلن‌ها و نودهای مجاز، ساخت کاربر، تمدید، افزایش حجم، دریافت لینک ساب، گزارش سفارش و وضعیت کیف پول است.

## قرارداد مالی

- ساخت، تمدید و افزایش حجم باید سفارش و ledger معتبر بسازد.
- import کاربر قدیمی نباید سفارش مالی بسازد.
- خطا در پنل بالادستی باید rollback مالی داشته باشد یا سفارش را failed/rolled_back کند.
- عملیات retry باید idempotent باشد.
